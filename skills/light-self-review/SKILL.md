# 自动反思与自我审查

## 工作方式（常驻，输出前必跑）
任何技能产出在交付前，先过一遍自检清单，发现问题就**先修后交**，不把粗糙结果直接抛给用户。重大产出可迭代多轮。

**证据闸门（借 verification-before-completion）**：不下「完成/通过/修好了」之类结论，除非当前轮就有新鲜证据。五步——①想清「什么命令/检查能证明它」②当场重跑(不引用旧结果) ③读完整输出、查 exit code、数失败 ④确认输出真支撑结论 ⑤才下结论。跳步=撒谎而非验证。红旗词：should/probably/seems to、提前自我满足、轻信子代理报告。
- 测试通过 = 有 0 失败的测试输出；构建成功 = exit 0（光过 lint 不够）；bug 修复 = 原始症状被测且通过；需求满足 = 逐条核对清单。
- **借口拦截**：临交付前大脑最会编借口跳过验证。「改动小不用跑/应该能过/上次跑过/先交了再说/审稿人不会注意」等都是借口而非判断，命中即回到证据闸门重跑。完整 26 条 excuse→reality 见 `references/excuse_intercept.md`。

**三态判定**：自检每一项都落到 通过✓ / 不通过✖ / 警告! 的明确判定（同 Deepchecks 的 check+condition、安全工具的 HIGH/MED/LOW/SAFE），不写含糊感受。不通过项附严重级 Critical(立即修)/Important(交付前修)/Minor(记录待办)。

## 通用自检清单
> 每项的最小 通过✓/不通过✖ 反例对照见 `references/self_check_contrasts.md`；可勾选清单见 `assets/self_review_checklist.md`。

□ **逻辑**：论证链有无断点、跳跃、自相矛盾？
□ **事实**：数据/引用/声称是否真实可核查？有无臆造(CONVENTIONS §4)？
□ **格式**：是否符合目标要求(模板/字数/引用风格/编号)？
□ **表达**：是否清晰、专业、无歧义、无口语？
□ **创新**：贡献是否突出、不是常规堆叠(联动 m04 视角)？
□ **引用**：是否准确、相关、不遗漏、不堆砌(联动 m10)？
□ **夸大**：结论是否超出证据支撑(慎用 novel/significantly)？
□ **审美**：图表/排版/PPT 风格是否统一(联动 a07)？
□ **重复**：有无冗余内容、重复段落？
□ **结构**：组织是否合理、层次是否清晰？
□ **可执行**：方案/代码/步骤是否真能跑通、能落地？

## 按产出类型补充
- **代码**：能否运行、有无测试、安全隐患(联动 a03 verification)。回归测试要走红-绿循环——撤掉修复必须失败、恢复后通过，「永不失败的测试什么都没证明」。新功能/改 bug 走 TDD：先写失败测试→看它真失败(不是 error)→写最简代码转绿→重构。依赖安全可用 SCA/能力分析视角自查：装包脚本、网络/shell 访问、混淆代码、typosquatting(仿冒名)；CLI 可借 `snyk test`(SCA)/`snyk code test`(SAST)/`sfw <装包命令>`(Socket 装前拦截)，severity-threshold 在 CI 降噪。若产出本身是 agent skill/MCP，按 skill-scanner/mcp-scanner 维度自查：prompt injection、数据外泄、含糊触发描述、跨技能重叠、.pyc 完整性、管道命令污点——并牢记这类扫描「无发现≠无威胁」，仍需人工复核。
- **论文/材料**：用同行评审维度自审——原创性、方法学是否扎实、**结论是否被结果支撑**(评审最经典失败点，多数人会漏)、文献根基、贡献是否对领域有用、格式合规(联动 m14)。科学主张再过批判性思维：独立 verify 源头、分清相关与因果、是否可证伪、举证责任在提出方、是解释「为什么」还是只描述「是什么」。
  - **核心撞车终检（交付前最后一道闸）**：论文定稿前，用**核心结论/核心方法**当关键词再检索一次（不是领域泛词，至少 2 库），专门找"最像的那一篇"。曾有项目做完整套实验、写完论文才发现核心已被前作发表，新颖性从自评 70 跌到 35-45。三态判定：✖ 核心撞车且论文未承认前作 = Critical，立即回 m08/m04 重定位（承认前作、改打广度/严谨/负结果牌，绝不假装首创）；! 撞车但已诚实承认并讲清 delta = 警告，确认 framing 是否已收缩到证据撑得住的尺度；✓ 无撞车且阴性证据充分。同时预演"审稿人会用什么理由拒"top-3，每条在论文里要么有应答、要么已通过缩小 claim 规避。这与 m03/m04 的核心撞车检查同源，是输出端兜底——上游漏了这里必须拦住。
- **数据/结果**：统计是否站得住(联动 m06)。对照/样本/漂移可借 Deepchecks(data_integrity / train_test_validation / model_evaluation 套件+condition)或 Evidently(DataDriftPreset，注意 run 参数是当前在前、基线在后)的检查清单逐项核。
- **合规**：有无伦理/版权/隐私风险(联动 a10)。审计当成带证据的逐条核查：定范围→逐条三态判定→不通过附修复建议与来源位置。
- **自评打分(对抗式批判)**：按 5 维给反馈打分——可操作性、具体性(指到具体元素)、论证(给证据非空断言)、是否提出明确解法、语气专业；并把致命问题排在格式小问题前面，别陷入「只挑表述/格式」的表面陷阱。
- **开场即上强度(grill 规则)**：自审**首句就直接列三个最致命弱点**，禁止"总体不错/大体到位"式客套开场与缓冲句——对自己的产出，缓冲就是放水。三个弱点按严重度排序，先于任何肯定性表述。

## 失败循环识别
若同一问题修了两次仍不对，停止打补丁，诊断根因，换思路(systematic-debugging)。根因调查：细读报错→稳定复现→查最近改动(git/依赖/配置)→多组件系统在边界加日志定位坏在哪层→逆向追数据流到源头，在源头修不在症状处修。**修≥3 次仍不行=架构问题**：停手，质疑架构(信号：每修一处就暴露新耦合/需大重构/按下葫芦浮起瓢)，与用户讨论后再动。

## 接收外部意见（借 receiving-code-review）
对评审/反馈不「表演式同意」、不「盲目实现」：先核验意见技术上是否成立、是否清晰，对可疑处提质疑或要澄清，只在理解+验证后才改；反驳要给代码/测试/技术理由，不空驳。
- **禁表演式同意**：「你说得太对了/马上改/好的好的」等空附和一律禁用，同意必须带证据，反驳也必须带证据。
- **不清楚就全停**：对意见含义/范围/目标有任何不确定，停下要澄清，不用猜测继续往下做。
- **范围纪律 + YAGNI**：只改被指出的问题，不借机重构无关代码；不为「将来可能用到」加抽象/配置/防御代码。
- 禁用措辞清单、标准流程、问题分级见 `references/receiving_feedback.md`。

## 产出
默认静默自检并直接交付修正后的结果；若发现无法自行解决的重大问题，明确告知用户问题与建议，不隐瞒。
